# Data Patches Ledger

Tracks places where the engine overrides the upstream DDOBuilderV2 / wiki data
because the source values don't match observed in-game behavior. Each entry
should eventually be filed upstream so the patch can be removed.

Each row describes:
- **Patch site** – the file + symbol where the override lives.
- **Symptom** – what the unpatched data produced.
- **Override** – what we apply instead.
- **Why** – the source of truth (in-game observation, wiki, etc.).
- **Upstream** – link or note for filing the bug; `—` if not yet filed.

## SLA charges (`src/engine/dps/slaCharges.ts::SLA_CHARGE_PATCHES`)

Some SLA-granting effects don't encode their per-rest charge count — the
typical `<Amount>0 0 0 0</Amount>` array leaves index 0 (charges) blank
even when the in-game ability is charge-limited.

Each entry matches a sourced SLA by a substring of the source label and
the spell name. Applied during SLA collection in `runEngine`; surfaced as
`MagicAbility.charges` for the rotation timeline to enforce.

| Source contains | Spell | Charges/rest | Why | Upstream |
|---|---|---|---|---|
| `Past Life: Arcane Initiate` | Magic Missile | 10 | In-game tooltip ("ten times per rest"). | — |

`charges = 0` means unlimited; the timeline only enforces a cap when this
value is > 0.

## Spell cooldowns (XML overlay via merge scripts)

Upstream `Spells.xml` never carries `<Cooldown>` on damaging spells — values
only appear (sometimes) in description prose. We maintain the cooldowns
out-of-band and inject them into the XML so the engine reads them like
any other field.

**Three files**:

- **`scripts/spell-cooldown-overrides.json`** — manually-verified values.
  Highest priority. Add a spell here whenever you confirm its cooldown
  in-game and want to pin it.
- **`scripts/spell-cooldowns.json`** — the merged, comprehensive source of
  truth that the merge script reads. Generated by the seeder; checked in.
- **`public/data/Spells.xml`** — the actual data file the engine reads.
  Gets `<Cooldown>N</Cooldown>` children injected.

**Two scripts**:

- **`node scripts/seedSpellCooldowns.mjs`** — regenerates
  `spell-cooldowns.json` from three sources, in priority:
  1. `spell-cooldown-overrides.json` (verified)
  2. `Cooldown:` regex mined out of each spell's `<Description>`
  3. Level-based default (lowest class level any class lists for the
     spell): cantrip/1/2 → 2s, 3 → 4s, 4/5 → 6s, 6 → 8s, 7 → 12s,
     8 → 15s, 9 → 30s.
- **`node scripts/applySpellCooldowns.mjs`** — idempotently injects every
  entry from `spell-cooldowns.json` into `public/data/Spells.xml`.

**Workflow**:

1. Edit `scripts/spell-cooldown-overrides.json` with verified values.
2. `node scripts/seedSpellCooldowns.mjs` → updates `spell-cooldowns.json`.
3. `node scripts/applySpellCooldowns.mjs` → rewrites `Spells.xml`.
4. Commit all three files.

**After an upstream Spells.xml refresh**: pull the new file, then re-run
both scripts. Cooldowns get reapplied without manual rework. The seeder
keeps the JSON sorted alphabetically for stable diffs and the apply
script is idempotent (it replaces, never duplicates).

Current state (post-seed): **201** damaging spells carry a `<Cooldown>`.
60 lack a class spell entry (SLA-only, NPC-only, etc.) and remain at the
parser default of 0 until manually overridden.

> ⚠ Most non-overridden values are LEVEL-BASED DEFAULTS — guesses, not
> verified. Treat them as starting points. As you confirm values in-game,
> add them to `spell-cooldown-overrides.json` and re-seed.

## Inline XML edits

Direct mutations to `public/data/EnhancementTrees/*.tree.xml`. Used when
the upstream encoding requires engine support we don't have yet — easier
to reshape the data into a form the existing engine handles than to add
a new evaluator path. Restore the original block on each upstream XML
refresh.

| File | Site | Symptom | Override | Why |
|---|---|---|---|---|
| `Rogue_ArcaneTrickster.tree.xml` | `Arcane Trickster: Sharp Magic` effect | Single `AType=Stacks` effect with `<Amount size="31">0 0 1 0 2 0 …</Amount>` indexed by metamagic-feat count — our engine looks up `amount[rankCount-1]=amount[0]=0` instead of `amount[2 × metamagicCount]`. | Replaced with 11 `AType=FeatCount` effects, one per metamagic feat (`Empower Spell`, `Maximize Spell`, …, `Eschew Materials`), each granting `+1 SneakAttackDice` when its feat is trained. Net behavior matches "+1 sneak die per metamagic feat trained" without needing a new evaluator. | The description says "+1 Sneak Attack dice for every Metamagic feat you've trained." Enhancement is unique enough that a one-off XML reshape is cheaper than implementing generic stack-source counting. |
| `Rogue_Mechanic.tree.xml` | `Mechanic: Improved Detection` (`MechCore4`) `SneakAttackDice` effect | The `<Requirement><Type>Stance</Type><Item>Ranged</Item></Requirement>` references a stance named `Ranged`, which doesn't exist in `Stances.xml` (the auto-stance for using a ranged weapon is `Ranged Combat`). Effect never fires. | Rename `<Item>Ranged</Item>` → `<Item>Ranged Combat</Item>` so the requirement matches the actual auto-stance name. | Description says "1 extra Sneak Attack die with bows, crossbows, and thrown weapons" — gating on ranged combat is correct; only the stance label was wrong. Likely an upstream typo. |
| `Spells.xml` (new entry) + `Shadowdancer.tree.xml` Nightmare Lance | Nightmare Lance has no `<Spell>` entry in `Spells.xml`, and the Shadowdancer enhancement marks itself `<Clickie/>` without `SpellLikeAbility>`. Net: the SLA never appears in the rotation palette. | (1) Add `<Spell><Name>Nightmare Lance</Name>…</Spell>` with the damage shape from the enhancement description (1d6+6 Force per CL, MaxCasterLevel 30, 8s Cooldown, Conjuration school). (2) Append a SpellLikeAbility effect to the EnhancementSelection. | Enhancement description says it's an Epic Strike SLA. |
| All Epic Strike clickies + an explicit allowlist (Hunt's End, Boulder's Might, Conjure Stone, Beguiling Charm) — `scripts/patchEpicStrikes.mjs` | Same shape as Nightmare Lance: `<Clickie/>` enhancement, no upstream `SpellLikeAbility>` wiring. | Run `node scripts/patchEpicStrikes.mjs` after each upstream tree refresh. The script walks every EnhancementSelection / EnhancementTreeItem and inserts a SpellLikeAbility effect when the description contains "Epic Strike" or the name is in `EXTRA_NAMES`. Cost + cooldown are parsed from description prose. | Idempotent (skips entries already wired). Tightly scoped on purpose: blanket-wiring every `<Clickie/>` would inflate the SLA list with passive feature toggles, weapon-attack augments, action boosts (which are charge-limited SLAs to be modeled later), summons, etc. that don't belong in the magic rotation palette. Add new names to `EXTRA_NAMES` as users surface them. |
| Engine-side cooldown grouping for Epic Strikes | Epic Strikes share a cooldown pool: firing any one puts every other Epic Strike onto its cooldown. Without grouping the engine treats them as independent CDs and a rotation can spam them in parallel. | `MagicAbility.cooldownGroup = 'epic-strike'` is set in `getMagicAbilities` whenever the SLA's source label matches `Epic Strike\s*→`. `resolveTimeline` and `findFirstAvailableSlot` in `src/engine/dps/timing.ts` track a `groupReady` map and clamp each cast's start time to `max(ownReady, groupReady)`. | Implemented in code, no XML patch required — the destiny trees use a uniform "Epic Strike → ChosenStrike" source pattern that's stable to detect against. |

## How to add a new patch

1. Pick the right place based on the data source:
   - Item buff (per-item-instance value override): `ITEM_BUFF_PATCHES`
   - Augment effect shape: `AUGMENT_EFFECT_PATCHES`
   - Enhancement tree effect: `ENHANCEMENT_TREE_PATCHES`
   - SLA per-rest charges: `SLA_CHARGE_PATCHES`
   - Spell cooldown: edit `scripts/spell-cooldowns.json`, run the merge script
2. Add the entry — keep the patch as narrow as possible (specific name,
   specific buff/type combination) so it doesn't accidentally rewrite
   future data.
3. Add a row to the matching table here. Fill in the Why with whatever
   evidence you have (screenshot path, wiki link, in-game observation).
4. Update snapshots: `npm test -- --run -u`.
5. When the upstream fix lands, remove the patch entry, the table row,
   and re-snapshot.
